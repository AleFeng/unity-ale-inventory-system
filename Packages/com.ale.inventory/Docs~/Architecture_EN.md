# Architecture

- Back to [documentation](../README_EN.md)

## Design Goals

The editor always and only works on ScriptableObjects; JSON / binary serve only as one-way export formats (consumed by runtime / builds). So the editor side needn't worry about serialization round-trip compatibility, and Undo/Redo only affects the SO.

---

## Core Data Model

### Flexible Attribute System

Uses a **tagged union (variant) + array combination**, rather than `[SerializeReference]`.

`AttributeValue` (`Runtime/Data/AttributeValue.cs`) carries:

- `EFieldType _type`: the current type (22 in total, see below);
- `bool _isArray`: scalar / array;
- `string _enumTypeRef`: the enum type name (Enum only);
- five type-categorized backing lists: `List<int>` / `List<float>` / `List<string>` / `List<Object>` / `List<AnimationCurve>`.

**Backing list conventions**:

| Type group | Backing list | Step |
|--------|---------|------|
| Bool / Enum | `ints` | 1 |
| Int | `ints` | 1 |
| VectorInt2 / 3 / 4 | `ints` | 2 / 3 / 4 |
| Float | `floats` | 1 |
| Vector2 / 3 / 4 | `floats` | 2 / 3 / 4 |
| Color | `floats` | 4 |
| String | `strings` | 1 |
| Text | `strings` | 3 (plain text + table reference + entry key) |
| Sprite / Prefab / Texture / Material / AudioClip / AnimationClip / PhysicsMaterial / PhysicsMaterial2D | `objRefs` | 1 |
| AnimationCurve | `curves` | 1 |

A scalar stores element `[0]`, an array stores `[0..n]` (flattened when step > 1).

Object-type fields have a string address list `_objAddresses` parallel to `objRefs` (carrying the Addressable address / AssetReference authorized GUID): with `IS_ADDRESSABLE` enabled, the editor authorizes via the native AssetReference selector (storing only the GUID, leaving the objRefs slot empty, so loading the database no longer loads the assets too); at runtime, via the `InventoryAssets` facade, "prefer the live reference, otherwise load the address asynchronously", auto-unloading when the host is destroyed. See [Attribute System – Asset Field Loading](AttributeSystem_EN.md#asset-field-loading-addressables) for details.

**Why not `[SerializeReference]`**: a tagged union serializes natively in SO, natively supports Undo/Redo, and needs no polymorphic PropertyDrawer; `[SerializeReference]` has risks of managed-reference/Undo corruption. The cost (a few empty lists per value) is negligible for config-time data.

### Stable Enum References

`EnumType` (`Runtime/Data/EnumType.cs`) maintains a monotonically increasing `nextValue`; it's assigned and incremented on `AddItem`, and not reclaimed on deletion. An attribute value stores the **enum value (int)**, not the index, so reordering enum items doesn't break existing references.

An enum item (`EnumItem`) can itself carry a group of `AttributeDefinition` custom attribute fields, used to describe the enum item's extra data (e.g. each class's base attribute bonuses).

### Items and Warehouses

```
InventoryDatabase (ScriptableObject)
├── List<EnumType>                  Enum type definitions
├── List<FunctionTag>               Function tag definitions (with attribute fields)
├── List<ItemTemplate>              Item templates (with attribute fields + locked function tag references)
├── List<Item>                      Item entries (with source template reference + own tag references + attribute values)
├── List<InventoryTemplate>         Warehouse templates (with config parameters + attribute fields)
├── List<Inventory>                 Warehouse entries (with source template reference + config parameter overrides + attribute values)
├── List<AttributeDefinition>       Sort options' optional extra attribute fields (schema; name/ignore IDs are now built into SortOption, this is empty by default)
├── List<SortOption>                Sort options (auto-generated by RebuildSortOptions; with built-in displayName(Text) + ignoreIds)
├── List<NumberFormatConfig>        Number format configs (referenced by name)
├── List<ShopTemplate> / List<Shop> Shop templates / shops (with product groups, products, refresh schedules)
├── Crafting: List<CraftingGroupTag> / List<CraftingBlueprintTemplate> / List<CraftingBlueprint>
└── Equipment: List<EquipmentGroupTag> / List<EquipmentGroupTemplate> / List<EquipmentGroup>
                                   (an equipment group contains slot lists → equipment slots + item limits + equipment attribute fields)
```

`Item.RebuildAttributes(db)` collects the expected fields by priority (template's own → template-locked tags → item's own tags), adding/removing/reordering the `values` list to keep it synced with the source definitions (idempotent, not affecting Undo history).

The `RebuildAttributes(db)` of `Inventory` / `Shop` / `CraftingBlueprint` / `EquipmentGroup` is similar to items, each collecting attribute fields only from its respective template (warehouse template / shop template / blueprint template / equipment-group template). `SortOption` is auto-synced by `InventoryDatabase.RebuildSortOptions` from the sort fields of all warehouse templates.

> An equipment group and an equipment-group template share `IEquipmentConfig` (equipment warehouses + slot lists + equipment attribute fields), with the template carrying all configurable options; creating an equipment group from a template deep-copies these configs, after which the equipment group is independently editable (the opposite of the Crafting System's "template-level read-only"). The equipment warehouses list specifies the warehouses the equipment system / UI can interact with; on unequip it finds the first warehouse with room from Index0.

## Data Flow

```
InventoryDatabase (SO)  ──edit──▶  still an SO
        │
        ├─ export ─▶ InventoryDtoMapper.ToDto ─▶ JsonUtility / BinaryWriter ─▶ .json / .bytes
        │                                   (Sprite and other references converted to GUIDs by EditorAssetGuidResolver)
        │
        └─ runtime load ◀─ InventoryJson/BinarySerializer.Import ◀─ .json / .bytes
                       (NullAssetRefResolver: object references kept empty;
                         Addressable mode: AddressableAssetRefResolver loads asynchronously by address)
```

The DTO layer (`Runtime/Serialization/InventoryDtoModels.cs`) is a flat structure that mirrors the data model one-to-one, the only difference being that object references are carried as GUID strings.

```
├── ItemSystemTab              Item System tab

├── InventorySystemTab         Warehouse System tab
│   ├── InventoryTemplatePanel Warehouse template list + edit panel
│   ├── InventoryListPanel     Warehouse list + InventoryInspectorPanel
│   └── (also SortOptionPanel / NumberFormatConfigPanel)
├── ShopSystemTab             Shop System tab
│   ├── ShopTemplatePanel      Shop template list + edit panel
│   ├── ShopListPanel          Shop list + ShopInspectorPanel (product groups / products)
├── CraftingSystemTab         Crafting System tab
│   ├── CraftingGroupTagPanel  Group tag list + edit panel
│   ├── CraftingTemplatePanel  Blueprint template list + edit panel
│   └── CraftingListPanel      Blueprint list + CraftingInspectorPanel
└── EquipmentSystemTab        Equipment System tab
    ├── EquipmentGroupTagPanel Group tag list + edit panel
    ├── EquipmentTemplatePanel Equipment-group template list + edit panel (name/color + shared config + custom attribute fields)
    └── EquipmentListPanel     Equipment-group list + EquipmentInspectorPanel (nested slot lists / equipment slots / item limits / attribute fields)
```

The "Skill System" tab is currently a placeholder (DrawStub), to be implemented in a later phase.

> The Equipment System's "slot list + equipment attribute fields" are drawn uniformly by `Editor/Common/EquipmentConfigDrawer` (reused by the equipment-group Inspector and the equipment-group template Inspector), and the drag-reordering of nested sub-lists is isolated via a `Dictionary<string, EditorReorderableDrag>` keyed by path.

### Common Drawers

| Class | Responsibility |
|----|------|
| `AttributeFieldDrawer` | Draws a single `AttributeValue` by `EFieldType` (dual implementation: GUILayout path + Rect path) |
| `AttributeDefinitionDrawer` | Draws the full edit panel of an `AttributeDefinition` (Rect-based, for ReorderableList) |
| `AttributeDefinitionListDrawer` | An attribute field definition list with drag-reordering (uses `ReorderableList` internally, drawElementCallback fully Rect-based) |

**Rect-based principle**: within `ReorderableList.drawElementCallback`, calling `GUILayout.BeginArea / GUI.BeginGroup` is strictly forbidden, otherwise a mismatch in GUILayout slot counts between Layout and Repaint throws a "Getting control X's position..." exception. All `DrawRect` methods use only `EditorGUI.*`.

### Unified Modification Flow

All edit operations follow: `ctx.RecordUndo(description) → modify data → ctx.MarkDirty()`.

Duplicate-ID detection is recomputed and cached (`HashSet<string>`) by `DuplicateIdChecker` in the Layout phase, at very low cost (O(n) per Layout event).

## Runtime Architecture

```
InventoryRuntimeManager (MonoBehaviour singleton)
├── InventoryDatabase[]    ─register→  InventoryDataManager (static definition queries)
└── Dictionary<inventoryId, RuntimeInventoryState>  ─maintain→  each warehouse's runtime slot list
```

`InventoryRuntimeManager` responsibilities:
- Registers the databases into `InventoryDataManager` on initialization;
- Creates an empty `RuntimeInventoryState` (slot list) for each defined warehouse;
- Provides the runtime operations `TryAddItem / TryRemoveItem / TryRemoveItemById / SortInventory`;
- Provides `GetSaveData / LoadSaveData` interfaces to integrate with the game's save system;
- Publishes the `OnInventoryChanged(inventoryId)` event for the UI layer to subscribe and refresh.

`InventoryDataManager` responsibilities (pure data query, stateless):
- Registers `InventoryDatabase` (supports multiple, merged queries);
- Queries `Item / Inventory / EnumType / FunctionTag / ItemTemplate / InventoryTemplate` by ID;
- Supports registering from three sources: `.asset`, JSON text, binary bytes.

### Subsystem Runtime Managers

The runtime logic of the shop and crafting systems is handled by two **lightweight singletons** (`InventorySystemSingleton<T>`, auto-created on first access, non-MonoBehaviour); neither holds catalog data itself (the catalog comes from registered databases), and warehouse reads/writes always go through `InventoryRuntimeManager`:

- `ShopRuntimeManager`: price resolution (multi-currency), currency / holdings tally across trade warehouses, resetting the tradeable count per the refresh schedule, purchase / recycle (auto-reducing the trade count by count / currency / capacity), **per-player trade progress save**; event `OnShopChanged`.
- `CraftingRuntimeManager`: craftable-count computation, deducting materials / placing output across crafting warehouses (executing one craft); **no state of its own, no save**, continuous crafting driven in a loop by the UI layer.
- `EquipmentRuntimeManager`: maintains equipped state as `equipment-group ID → (slot ID → equipped item ID)`; equip / unequip / swap cooperates with `InventoryRuntimeManager` to move items (rolling back when the old item can't be returned), limit matching is **all-AND**, auto slot-finding, summing across equipped items per the "equipment attribute field list" for total bonuses; **has an equipped-state save** (`GetSaveData` / `LoadSaveData`); event `OnEquipmentChanged`.

The three clocks needed for shop refresh (game / local / server time) are registered via `InventoryRuntimeManager.RegisterTimeGetter`, falling back to system local time when unregistered.

## Extension Guide

When adding a new subsystem (skill, etc.) (the shop / crafting / equipment systems are implemented per this pattern and can serve as references):

1. Add the corresponding data list in `InventoryDatabase` (+ getter / `CloneFrom` / `Validate`);
2. Create a subfolder under `Editor/`, implementing the three-column panels (reusing `AttributeDefinitionListDrawer` and `AttributeFieldDrawer`);
3. Register the new tab in `InventoryEditorWindow` (+ duplicate-ID scanning / `RebuildAllAttributes`);
4. Add the corresponding query method in `InventoryDataManager`; runtime logic is handled by a lightweight singleton (`InventorySystemSingleton<T>`);
5. (Optional) add a DTO mirror in `InventoryDtoModels.cs` — the shop / crafting / equipment systems don't, because the runtime directly consumes the registered `InventoryDatabase` (ScriptableObject), and DTO export currently covers only the Item System data.

The attribute system (`AttributeValue / AttributeDefinition / EFieldType`), enum types, function tags, and the DTO serialization framework can all be reused directly.
